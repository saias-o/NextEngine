01. Getting Started with NextEngine

NextEngine is designed around a component-hierarchical architecture resembling modern engines like Godot or Unity, but optimized for minimal overhead, direct C++ execution, and native Large Language Model (LLM) interaction.

To make NextEngine familiar to game creators, here is how its core structural components map to Unity conventions:

NextEngine C++ Type Unity C# Equivalent Function / Purpose
ne::Node GameObject The fundamental object in the scene hierarchy. Contains transform and children.
ne::Behaviour MonoBehaviour The base scripting component for attaching custom game logic and lifecycle tick callbacks.
ne::Transform Transform Struct containing position (vec3), rotation (quat), and scale (vec3) parameters.
ne::Input Input Static utility class to query keyboard button states and mouse deltas synchronously.
ne::MeshNode MeshRenderer + MeshFilter A subclass node that references an asset mesh and material descriptor to draw 3D models.
onDrawInspector() OnInspectorGUI() Virtual function override to draw custom ImGui variable sliders and debug properties.

Rather than wrapping engine compilation in opaque binaries, NextEngine provides developers with immediate C++ compiler access. Game modules are compiled as dynamic link libraries (DLLs) which the core editor hot-swaps, allowing quick iteration loops.

Architecture Tip: The core scene graph is purely data-oriented. Nodes contain matrices, translations, and hierarchy relations. All Vulkan resource uploads and draws are driven by the central engine loop querying mesh nodes.

02. ne::Node Reference

`ne::Node` is the base class for all elements in the scene hierarchy. It handles parent-child relations and computes recursive local-to-world transform transformations.

Below is the active class declaration extracted directly from src/scene/Node.hpp:

scene/Node.hpp (C++ Header) 
namespace ne {
 
class Node {
public:
    explicit Node(std::string name = "Node");
    virtual ~Node();
    Node(const Node&) = delete;
    Node& operator=(const Node&) = delete;

    // Takes ownership of `child` and sets its parent. Returns borrowed raw pointer.
    Node* addChild(std::unique_ptr<Node> child);
    
    // Find the unique_ptr for `child`, erase it (which deletes it), and return true.
    bool removeChild(Node* child);
    
    // Detach a child from this node and return ownership unique_ptr.
    std::unique_ptr<Node> detachChild(Node* child);

    // Construct a child of type T in-place. Automatically forwards constructor args.
    template <typename T, typename... Args>
    T* createChild(Args&&... args);

    // Attach behaviours for scripting
    template <typename T, typename... Args>
    T* addBehaviour(Args&&... args);
    Behaviour* addBehaviour(std::unique_ptr<Behaviour> behaviour);

    // Transforms & Hierarchy
    void updateTransforms(const glm::mat4& parentWorld, bool parentDirty);
    const glm::mat4& worldTransform() const;
    Transform& transform();
    Node* parent() const;
    const std::vector<std::unique_ptr<Node>>& children() const;
    
    // Depth-first scene tree traversal
    void traverse(const std::function<void(Node&, const glm::mat4&)>& visit);
};

} // namespace ne

03. ne::Scene Reference

A `ne::Scene` represents the database of the current world. It inherits from `ne::Node` as the root of the hierarchy tree, but manages flattened collections of meshes and lights for high-speed Vulkan rendering.

Below is the class definition from src/scene/Scene.hpp:

scene/Scene.hpp (C++ Header) 
namespace ne {

struct SceneSettings {
    glm::vec4 ambientLight{0.04f, 0.04f, 0.05f, 0.0f};
    glm::vec4 clearColor{0.1f, 0.1f, 0.1f, 1.0f};
    bool enablePostProcessing = true;
};

class Scene : public Node {
public:
    Scene();

    // Invokes updateTree on the root, recursively updating all behaviours
    void update(float dt);

    SceneSettings& settings();
    const std::vector<MeshNode*>& meshes() const;
    const std::vector<LightNode*>& lights() const;
};

} // namespace ne

04. ne::Behaviour Reference

`ne::Behaviour` is the base class for custom scripting (analogous to Unity's MonoBehaviour). Behaviours hook into the node lifecycle methods `onReady` and `onUpdate`.

scene/Behaviour.hpp (C++ Header) 
namespace ne {

class Behaviour {
public:
    virtual ~Behaviour() = default;
    
    // Lifecycle hooks
    virtual void onReady() {}
    virtual void onUpdate(float dt) {}
    virtual void onDrawInspector() {}

    // Serialization for saving/loading scenes
    virtual const char* typeName() const { return nullptr; }
    virtual void save(nlohmann::json&) const {}
    virtual void load(const nlohmann::json&) {}

    Node* node() const { return node_; }
};

} // namespace ne

05. ne::ResourceManager Reference

`ne::ResourceManager` handles loading and caching of Vulkan textures, meshes, and materials. It ensures assets are uploaded to GPU memory only once and are referenced via unique GUIDs.

Below is the resource management structure from src/graphics/ResourceManager.hpp:

graphics/ResourceManager.hpp (C++ Header) 
namespace ne {

class ResourceManager {
public:

    VkDescriptorSet allocateMaterialSet(VkDescriptorSetLayout layout);

06. ne::EditorUI & Panel Architecture

NextEngine features a modular, dockable Editor interface using Dear ImGui. The editor is structured into separate panel files to avoid large class coupling, allowing humans and LLM agents to easily maintain and extend visual panels.

Editor UI Layout & Panel Guides (For Humans)

When you launch the NextEngine editor, it lays out several dockable panels on the screen to manage your game state. Here is a description of how to use each panel:

  • Scene Hierarchy Panel (Left) : Lists all objects currently active in your scene in a tree structure. You can click objects to select them, double-click to rename them, and drag-and-drop nodes to change their parent-child relationships. Right-clicking a node allows you to create new children or delete the node.
  • Inspector Panel (Right) : Shows detailed properties of the selected node. It displays local transform coordinates (Position, Rotation, Scale inputs) and lists all attached components (`ne::Behaviour`). If the node is a light source or a mesh, it provides settings to change light colors or select mesh models.
  • File Browser Panel (Bottom) : A built-in filesystem navigator showing your project's assets directory. You can double-click folders to browse, search files, and drag assets (like .scene or .obj models) directly into the viewport or scene tree to load them.
  • Viewport Panel (Center) : The 3D Vulkan canvas rendering your game. It features tabs at the top to toggle between Scene Mode (where you can view and translate objects using translation/rotation gizmos) and Play Mode (which executes behaviour script logic in real-time).
  • Menu Bar Panel (Top) : The top file menu bar. Allows saving the scene, loading projects, compiling the game, opening project settings, or toggling the visibility of specific panels.

Below is the class definition of the coordinator from src/editor/EditorUI.hpp:

editor/EditorUI.hpp (C++ Header) 
namespace ne {

class EditorUI {
public:
    EditorUI();
    
    // Draw the full dockable editor layout
    void draw(Scene* scene, Camera* camera, Project* project, ResourceManager* resources, float dt);

    Node* selectedNode() const;
    bool isPlayMode() const;
    void setPlayMode(bool play);
    bool quitRequested() const;
    
    bool isViewportHovered(float mx, float my) const;

private:
    // Deferred operations for C++ memory safety (avoids iterator invalidation)
    Node* nodeToDelete_ = nullptr;
    Node* nodeToCreateChildUnder_ = nullptr;
    CreateNodeType createType_ = CreateNodeType::None;
    Node* nodeToReparent_ = nullptr;
    Node* newParent_ = nullptr;

    // Viewport and Gizmos
    GizmoMode gizmoMode_ = GizmoMode::Translate;
    GizmoAxis grabbedAxis_ = GizmoAxis::None;
    glm::vec2 viewportPos_{0.0f};
    glm::vec2 viewportSize_{0.0f};

    // Undo/Redo command stack
    CommandHistory history_;
    std::string clipboard_; // JSON string of a copied node subtree
};

} // namespace ne
Panel Splitting: The editor viewport is a transparent passthrough overlay drawn directly on top of the Vulkan rendering canvas. Scene hierarchy tree, inspector, and file system browser are rendered as dockable ImGui panel modules: SceneHierarchyPanel, InspectorPanel, FileBrowserPanel.

07. ne::Command (Undo/Redo System)

NextEngine implements a command history stack for undo and redo actions in the Editor. Every modification (moving nodes, renaming, deleting, spawning nodes) is wrapped in a subclass of `ne::Command` that keeps track of the delta state.

Below is the C++ interface definition from src/editor/Command.hpp:

editor/Command.hpp (C++ Header) 
namespace ne {

// Base interface for undoable editor operations
class Command {
public:
    virtual ~Command() = default;
    virtual void execute() = 0;
    virtual void undo() = 0;
    virtual const char* name() const { return "Command"; }
};

// Spawn a new node
class AddNodeCommand : public Command {
public:
    AddNodeCommand(Node* parent, std::unique_ptr<Node> node);
    void execute() override;
    void undo() override;
};

// Re-parent a node (via scene tree drag & drop)
class ReparentNodeCommand : public Command {
public:
    ReparentNodeCommand(Node* node, Node* newParent);
    void execute() override;
    void undo() override;
};

// Modify local transform properties
class TransformCommand : public Command {
public:
    TransformCommand(Node* node, const Transform& oldT, const Transform& newT);
    void execute() override;
    void undo() override;
};

} // namespace ne

08. LLM Native Agent Guide

NextEngine is engineered natively for autonomous Large Language Models (LLMs) and coding agents. Instead of forcing agents to click UI buttons, the engine exposes clean text-based communication sockets and JSON files.

1. Scene tree Inspection

AI agents can query the entire hierarchy of the world in a clean, structured JSON format rather than parsing complex binary buffers. A typical scene descriptor exported by SceneSerializer looks like:

scene_serialization.json 
{
  "scene": {
    "name": "MySampleWorld",
    "ambientLight": [0.04, 0.04, 0.05, 0.0],
    "nodes": [
      {
        "name": "MainCube",
        "type": "MeshNode",
        "transform": {
          "position": [0.0, 1.0, 0.0],
          "rotation": [1.0, 0.0, 0.0, 0.0],
          "scale": [2.0, 2.0, 2.0]
        },
        "behaviours": [
          {
            "type": "OrbitalMovement",
            "properties": { "speed": 1.5 }
          }
        ]
      }
    ]
  }
}

2. TCP Socket Command Protocol

Running the engine starts a local TCP listener on port localhost:8090. An agent can connect and send command payloads as JSON packets to manipulate the runtime scene state directly:

socket_payload.json 
{
  "command": "spawn_entity",
  "template": "mesh_voxel_cube",
  "name": "AgentVoxel",
  "position": [1.5, 0.0, -3.0],
  "script": "rotate_y.lua"
}

3. Hot-Reload Scripting Loop

To let LLM agents test C++ files and Lua scripts without closing the engine:

  • Agents write code to src/game/ files.
  • Agents submit a compile/reload command via the terminal loop or the TCP socket.
  • The engine hot-reloads Lua scripts, compiles C++ game logic DLLs, and swaps them dynamically at runtime.

09. Creating Your First Game

Building a game in NextEngine starts by creating behavior logic scripts (by subclassing `ne::Behaviour`) and attaching them to nodes in the scene tree hierarchy.

Follow this step-by-step example to create a rotating spinner object:

Step 1: Write the C++ Script Logic

Create a header file for your logic under src/game/Spinner.hpp. Override onReady() to initialize state, and onUpdate(dt) to animate coordinates every frame:

src/game/Spinner.hpp (C++ Header) 
#include "scene/Behaviour.hpp"
#include "scene/Node.hpp"

class Spinner : public ne::Behaviour {
public:
    float rotationSpeed = 2.0f; // Radians per second

    // Executed once when the node enters the scene tree
    void onReady() override {
        node()->setName("VoxelSpinner");
    }

    // Executed every frame
    void onUpdate(float dt) override {
        // Query the parent node's transform
        auto& t = node()->transform();
        
        // Rotate around Y axis
        t.rotation = glm::rotate(t.rotation, rotationSpeed * dt, glm::vec3(0.0f, 1.0f, 0.0f));
    }

    // Set stable type identification for scene saving
    const char* typeName() const override { return "Spinner"; }
};

Step 2: Attach the Behaviour to a Scene Node

In your main game initialization scene setup, create a new 3D mesh node and attach your behaviour script component:

src/game/GameInitializer.cpp (C++ Logic) 
#include "scene/Scene.hpp"
#include "scene/MeshNode.hpp"
#include "game/Spinner.hpp"

void initializeGame(ne::Scene* scene, ne::ResourceManager* resources) {
    // 1. Create a child mesh node under the scene root
    auto* spinnerNode = scene->createChild<ne::MeshNode>("MyRotatingMesh");
    
    // 2. Configure mesh and material
    auto meshId = resources->getOrRegister("models/crystal.obj");
    spinnerNode->setMesh(resources->getMesh(meshId));
    
    // 3. Attach custom spinner logic
    spinnerNode->addBehaviour<Spinner>();
}

10. Handling Inputs and Player Movement

In NextEngine, keyboard and mouse inputs are managed through the ne::Input system (`src/core/Input.hpp`). The input manager queries the low-level GLFW buffer once per frame, offering a clean, synchronous state access.

Below is a complete implementation of a keyboard-controlled Player movement script:

src/game/PlayerController.hpp (C++ Header) 
#include "scene/Behaviour.hpp"
#include "scene/Node.hpp"
#include "core/Input.hpp"
#include <GLFW/glfw3.h> // Keyboard key codes

class PlayerController : public ne::Behaviour {
public:
    float speed = 5.0f;
    glm::vec3 movementBounds{10.0f, 0.0f, 10.0f};

    void onUpdate(float dt) override {
        glm::vec3 direction{0.0f};

        // Query key states using ne::Input
        if (ne::Input::keyDown(GLFW_KEY_W)) direction.z -= 1.0f; // Forward
        if (ne::Input::keyDown(GLFW_KEY_S)) direction.z += 1.0f; // Backward
        if (ne::Input::keyDown(GLFW_KEY_A)) direction.x -= 1.0f; // Left
        if (ne::Input::keyDown(GLFW_KEY_D)) direction.x += 1.0f; // Right

        // Translate the node
        if (glm::length(direction) > 0.0f) {
            direction = glm::normalize(direction);
            auto& t = node()->transform();
            
            t.position += direction * speed * dt;
            
            // Constrain positions to bounds
            t.position.x = glm::clamp(t.position.x, -movementBounds.x, movementBounds.x);
            t.position.z = glm::clamp(t.position.z, -movementBounds.z, movementBounds.z);
        }
    }

    const char* typeName() const override { return "PlayerController"; }
};
Input Consistency: Querying input via ne::Input instead of direct GLFW calls prevents differences in input detection between multiple behaviors updating in the same frame.

11. Loading Assets & Material Management

NextEngine is built on Vulkan 1.2 which requires uploading vertices and textures to GPU buffers before drawing. The `ne::ResourceManager` automates this process by caching assets and binding them to shaders.

Loading a 3D Mesh and Applying Textures

To load character assets, register their relative filesystem paths. The resource manager checks for duplicate files, reads formatting data, uploads vertices, and returns cached pointers:

src/game/AssetLoader.cpp (C++ Logic) 
#include "graphics/ResourceManager.hpp"
#include "scene/MeshNode.hpp"

ne::MeshNode* spawnTexturedModel(ne::Scene* scene, ne::ResourceManager* resources) {
    // 1. Register asset files (returns GUID hashes)
    ne::AssetID meshHash = resources->getOrRegister("models/knight.obj", ne::AssetType::Mesh);
    ne::AssetID textureHash = resources->getOrRegister("textures/knight_color.png", ne::AssetType::Texture);

    // 2. Fetch or allocate material (Descriptor Set layout 1 bind)
    // Passes texture hash and base diffuse color vec4 multiplier
    ne::Material* material = resources->getMaterial(textureHash, glm::vec4(1.0f));

    // 3. Create a scene node and assign loaded materials
    auto* knightNode = scene->createChild<ne::MeshNode>("KnightPlayer");
    knightNode->setMesh(resources->getMesh(meshHash));
    knightNode->setMaterial(material);
    
    // Scale and place model in scene space
    knightNode->transform().scale = glm::vec3(0.5f);
    knightNode->transform().position = glm::vec3(0.0f, 0.0f, -5.0f);

    return knightNode;
}

12. Controlling the Camera

NextEngine uses the ne::Camera class (`src/core/Camera.hpp`) to compute view-projection matrices. To create a fly-camera control mechanism, you update camera positions using direction vectors derived from mouse movements.

The yaw/pitch angles represent the Euler orientation. The camera converts these angles into unit vectors:

  • front(): The direction vector pointing straight ahead.
  • right(): The direction pointing to the right of the camera viewport.
  • up(): The direction pointing directly above the camera.

Implementing a Dynamic Fly Camera Controller

This script controls the rendering camera, allowing the developer/player to look around by holding the right mouse button and move using the WASD keys:

src/game/FlyCameraController.hpp (C++ Header) 
#include "scene/Behaviour.hpp"
#include "core/Camera.hpp"
#include "core/Input.hpp"
#include <GLFW/glfw3.h>

class FlyCameraController : public ne::Behaviour {
public:
    float speed = 8.0f;
    float mouseSensitivity = 0.15f;
    ne::Camera* activeCamera = nullptr; // Set during scene init

    void onUpdate(float dt) override {
        if (!activeCamera) return;

        // 1. Mouse rotation (Only when holding Right Mouse Button)
        if (ne::Input::mouseButtonDown(GLFW_MOUSE_BUTTON_RIGHT)) {
            glm::vec2 mouseDelta = ne::Input::mouseDelta();
            
            // Adjust camera Euler angles
            activeCamera->rotate(
                mouseDelta.x * mouseSensitivity, 
                -mouseDelta.y * mouseSensitivity
            );
        }

        // 2. Keyboard travel relative to facing direction
        glm::vec3 travelDirection{0.0f};
        if (ne::Input::keyDown(GLFW_KEY_W)) travelDirection += activeCamera->front();
        if (ne::Input::keyDown(GLFW_KEY_S)) travelDirection -= activeCamera->front();
        if (ne::Input::keyDown(GLFW_KEY_A)) travelDirection -= activeCamera->right();
        if (ne::Input::keyDown(GLFW_KEY_D)) travelDirection += activeCamera->right();

        if (glm::length(travelDirection) > 0.0f) {
            activeCamera->position += glm::normalize(travelDirection) * speed * dt;
        }
    }

    const char* typeName() const override { return "FlyCameraController"; }
};

13. Lighting and Realtime Shadows

Illumination in NextEngine is driven by the ne::LightNode components (`src/scene/LightNode.hpp`). The engine supports both Directional (sunlight-style) and Point (local light bulb-style) light sources:

  • ne::LightType::Directional: Emits parallel rays throughout the scene. Direction is computed from the node's local vector (rotated by world transform matrix).
  • ne::LightType::Point: Emits light outwards in all directions from the node position. Uses the node transform coordinates and the range limit parameters.

Configuring Scene Lights & Shadow Casters

The example below shows how to add directional lighting and point light sources to the scene, and how to enable shadow casting for individual meshes:

src/game/LightSpawner.cpp (C++ Logic) 
#include "scene/Scene.hpp"
#include "scene/LightNode.hpp"
#include "scene/MeshNode.hpp"

void configureSceneLighting(ne::Scene* scene, ne::MeshNode* playerNode) {
    // 1. Create a sun-like Directional Light
    auto* sunNode = scene->createChild<ne::LightNode>("Sun", ne::LightType::Directional);
    sunNode->color = glm::vec3(1.0f, 0.98f, 0.92f); // Warm sun
    sunNode->intensity = 3.0f;
    sunNode->direction = glm::normalize(glm::vec3(-0.3f, -1.0f, -0.4f));

    // 2. Create a glowing Point Light near an object
    auto* pointLight = scene->createChild<ne::LightNode>("Torch", ne::LightType::Point);
    pointLight->color = glm::vec3(1.0f, 0.5f, 0.1f); // Orange fire glow
    pointLight->intensity = 5.0f;
    pointLight->range = 8.0f;
    pointLight->transform().position = glm::vec3(2.0f, 1.5f, -3.0f);

    // 3. Enable Shadow Casting on the player mesh
    if (playerNode) {
        playerNode->castShadows() = true; // Enables shadow map pass for this mesh
    }
}

14. Custom Inspector UI

When building gameplay scripts in NextEngine, you can create custom visual parameters in the editor by overriding the onDrawInspector() method. The editor's InspectorPanel calls this function inside the drawing loop.

Because the editor embeds Dear ImGui directly, you have access to a rich set of GUI widgets to write sliders, buttons, inputs, and checkbox controls to debug values in real-time.

Creating Slider and Debug Inputs

Below is a script showing how to expose variables and create trigger buttons:

src/game/EnemyAI.hpp (C++ Header) 
#include "scene/Behaviour.hpp"
#include <imgui.h> // Expose Dear ImGui calls

class EnemyAI : public ne::Behaviour {
public:
    float detectionRange = 12.0f;
    int health = 100;
    bool isAggressive = false;

    void onUpdate(float dt) override {
        // AI logic goes here...
    }

    // Overridden to customize what shows in the editor inspector panel
    void onDrawInspector() override {
        ImGui::Text("AI Configurations");
        ImGui::Separator();

        // 1. Slider widget for detection floating point
        ImGui::SliderFloat("Detection Range", &detectionRange, 2.0f, 30.0f);

        // 2. Checkbox input for aggressive toggles
        ImGui::Checkbox("Aggressive Mode", &isAggressive);

        // 3. Simple input for integer parameters
        ImGui::InputInt("Health Status", &health);

        // 4. Interactive button executing instantly
        if (ImGui::Button("Respawn / Heal Enemy")) {
            health = 100;
            isAggressive = false;
        }
    }

    const char* typeName() const override { return "EnemyAI"; }
};

15. Compilation & Standalone Build

NextEngine is structured around a standard CMake configuration pipeline (`CMakeLists.txt`), compiling your game assets and engine runtime into optimized machine binaries.

Building Standalone Executables via Command Line

cmake -B build -DCMAKE_BUILD_TYPE=Release

# 2. Compile all engine libraries and game scripts
cmake --build build --config Release

Packaging Assets for Shipping

To ship your final standalone game to players, distribute the following files from your output build directory:

  • assets/ folder (Containing your scene JSONs, textures, and 3D models)
